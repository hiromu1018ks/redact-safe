## 1. プロジェクト概要

### 1.1 背景

行政機関においては、情報公開請求・開示請求への対応として、個人情報・機密情報を含む文書の一部を黒塗り（墨消し）処理したうえで開示する業務が日常的に発生する。

現行の運用では、以下の問題が指摘されている。

- 手作業によるPDF編集は属人的かつ時間がかかる
- 単純な黒色矩形の重ね描きでは、PDF内部にテキストデータが残存し、コピー操作により情報が復元できる危険がある
- 開示前の確認作業が担当者の目視に依存しており、漏洩リスクが排除しきれていない

### 1.2 目的

本アプリケーションは、行政職員が安全かつ効率的に文書の黒塗り処理を行うことを支援するデスクトップアプリケーションである。

**最重要設計原則**: 「編集可能な仮マスキング（可逆）」と「不可逆な確定マスキング（完全削除）」の2段階設計により、実務フローへの適合と情報セキュリティの両立を実現する。

### 1.3 適用範囲

- 情報公開・個人情報開示請求への対応業務
- 行政内部文書の外部提供時における秘匿処理
- 会議録・議事録の公開版作成

### 1.4 対象外

- クラウドサービスへのアップロード・外部送信機能
- 文書管理システム（DMS）との統合（将来拡張）
- 画像ファイル（JPEG/PNG/TIFF等）の**直接**編集

> **補足（画像ファイルの入力方針）**: 本アプリケーションは **PDFのみを入力として受け付ける**。スキャン画像（JPEG/PNG/TIFF等）を処理したい場合は、事前に印刷・スキャナソフト等でPDFに変換してから本アプリに読み込むこと。アプリ内での画像→PDF自動変換は行わない。この方針を現場導入時に必ず周知すること。

> **画像→PDF変換時の品質条件**: 外部ツールで変換する場合、以下の条件を満たすこと。
> - 解像度: 300dpi以上でPDF化すること（150dpi以下はOCR精度が著しく低下）
> - ページ順: 元文書のページ順を維持すること
> - 圧縮: 可逆圧縮（FlateDecode）を推奨。JPEG再圧縮は品質95%以上とすること
> - 推奨ツール: Windows標準の「Print to PDF」、または導入機関が指定するPDF変換ツール
> - 1ファイル1文書: 複数画像を1PDFにまとめる際は、ページ順の整合性を確認すること

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| **仮マスキング** | UIレイヤ上に黒矩形を重ね表示するのみで、PDF本体のテキストデータは変更しない、編集可能な状態のマスキング |
| **確定マスキング** | PDFを画像化し黒塗りを焼き込むことで、テキストデータを完全に消去する不可逆処理 |
| **Redaction（黒塗り）** | 文書内の特定情報を不可逆的に消去・秘匿する処理。単なる視覚的隠蔽とは区別される |
| **bbox** | Bounding Box。テキスト・領域の矩形座標（x, y, 幅, 高さ）を表す。座標系の詳細は付録Bを参照 |
| **PII** | Personally Identifiable Information（個人識別情報）。氏名・住所・電話番号・マイナンバーなど |
| **機密情報** | PII以外に行政機関が公開を制限する情報。内部決裁情報・法人番号・職員番号・案件番号等。本書では「自動検出対象」と「手動確認対象」に分類する（Section 5.3参照） |
| **OCR** | Optical Character Recognition（光学文字認識）。画像化されたPDFからテキストを抽出する技術 |
| **レイアウト解析** | PDFページを段落・表・図・ヘッダ等の領域に分類する処理。文字認識の前処理として実行する |
| **ウォーターマーク** | 仮マスキング状態であることを示す警告的透かし文字 |
| **安全PDF** | 確定マスキング処理済みであり、テキスト復元が不可能な状態のPDF |
| **hidden data** | PDFのメタデータ・添付ファイル・フォームフィールド・注釈・JavaScript等、可視テキスト以外に含まれる隠れたデータの総称 |
| **hidden data サニタイズ** | 確定処理において可視テキストの黒塗りとは別に実施する、hidden data の完全除去処理 |
| **PDF point** | PDFの内部座標単位。1 point = 1/72 インチ。A4縦は 595.28 × 841.89 points |
| **編集者ロール** | 仮マスキングの作成・編集を担当する職員（担当者） |
| **確認者ロール** | 仮マスキング内容を確認・承認する管理職または別担当者 |
| **確定実行者ロール** | 確定マスキング処理を実行する権限を持つ職員（確認者と兼任可） |
| **状態遷移制約** | ドキュメントの処理状態（draft/confirmed/finalized）に応じて、実行可能な操作をアプリが制限する仕組み |

## 3. システム概要

### 3.1 システム構成概念

```
[入力PDF]
    ↓
[レイアウト解析]  ← PaddleOCR Layout Analysis（段落・表・図・ヘッダ領域を分類）
    ↓
[文字認識（OCR）]  ← PaddleOCR 認識モジュール / フォールバック: Tesseract
    ↓
[bbox出力]  ← テキスト要素ごとの座標・文字列を構造化（付録B参照）
    ↓
[個人情報・機密情報 自動検出]  ← 正規表現（自動検出対象）+ MeCab形態素解析
    ↓
[仮マスキング生成（フェーズ1）]  ← UIオーバーレイ（編集可能）
    ↓
[ユーザーによる確認・修正]  ← ON/OFF・位置調整・手動追加（手動確認対象も含む）
    ↓
[確認者による承認]  ← 確認者ロールによる最終確認
    ↓
[最終プレビュー確認]  ← 確定前の確認画面
    ↓
[確定マスキング処理（フェーズ2）]  ← 画像化→黒塗り焼き込み
    ↓
[hidden data サニタイズ]  ← メタデータ・注釈・添付・フォーム・JS を除去
    ↓
[安全PDF出力]  ← テキスト完全削除・hidden data 除去済み
    ↓
[ログ保存]
```

### 3.2 2段階マスキングの必要性

PDFの黒塗りには技術的に2種類の実装が存在する。

**危険な実装（非推奨）**：黒色矩形をテキストの上に重ね描きするのみ。テキストデータはPDF内部に残存し、テキスト選択・コピー操作によって情報が復元可能である。

**安全な実装（必須）**：テキストデータをPDFから完全に削除する。これが真の黒塗りであり、行政文書の開示における法的要件を満たすものである。

一方で、実務上は「自動検出結果を人間が確認・修正してから確定する」という運用フローが不可欠であるため、確認段階では編集可能な仮状態を維持し、公開前の最終操作で初めて不可逆処理を実行する設計とする。

## 4. 利用者・ロール定義・利用シナリオ

### 4.1 ロール定義と権限

本アプリは**単独作業**ではなく、**複数人の確認フロー**を前提とする。以下の3ロールを定義し、誰がどの操作を実行できるかを明示する。

#### 4.1.1 権限制御の2層分離

権限制御は以下の2層で構成し、**状態制約を常に優先**する。ロール権限と状態制約が矛盾する場合は、状態制約が上書きする。

- **ロール権限（第1層）**: その人が理論上持つ操作権限。ロールが上位であれば下位ロールの全操作を含む。
- **状態制約（第2層・優先）**: ドキュメントの現在状態に応じた実行可能操作。ロール権限を上書きして制限する。

> **設計原則**: 実装時は状態制約を最初に評価し、次にロール権限を評価すること。UI・バックエンドの双方で同一の判定ロジックを適用し、「UIでは操作できるのにバックエンドで弾かれる」齟齬を防ぐこと。

#### 4.1.2 ロール定義

| ロール | 説明 | ロール権限（理論上の操作範囲） |
|-------|------|-------------|
| **編集者** | 仮マスキングを作成・編集する担当職員 | PDFを開く / OCR実行 / 仮マスキングのON・OFF・追加・削除・位置調整 / 設定ファイルの保存 |
| **確認者** | 仮マスキング内容を確認・承認する管理職または別担当 | 編集者の全操作に加え、仮マスキングの確認承認（`confirmed`状態への遷移）・差し戻し |
| **確定実行者** | 確定マスキング処理を実行する権限を持つ職員 | 確認者の全操作に加え、「確定して出力」ボタンの押下（`finalized`状態への遷移）|

> **運用上の注意**: 確認者と確定実行者は兼任可能とするが、**編集者と確定実行者の兼任は推奨しない**。「マスキングを作った本人が確定する」という運用では、確認漏れが発見されにくくなるため、可能な限り別担当者が確認を行うこと。

#### 4.1.3 初期フェーズの本人識別（暫定案）

初期バージョンではアプリ内認証（ログイン機能）は実装せず、以下の暫定方式により本人識別を行う。将来バージョンでRBACを実装する予定（Section 18参照）。

- **OSログイン名**を操作者のデフォルト識別子として使用する
- **確認承認・確定実行時に操作者名の再入力ダイアログを表示**し、無操作での確定を防止する
- 再入力された操作者名はOSログイン名と共に監査ログおよび設定ファイルに記録する
- `confirmed_by` / `finalized_by` には**OSログイン名（不変）** を保存し、`confirmed_by_display` / `finalized_by_display` に再入力された表示名を併記する
- 編集者と確定実行者のOSログイン名が同一の場合、**アプリレベルで警告を表示**する（兼任抑止）

### 4.2 処理状態遷移

ドキュメントは以下の3状態を持ち、各状態で実行可能な操作が**状態制約（第2層・優先）**により制限される。ロール権限（第1層）で許可されていても、状態制約が禁止する操作は実行できない。

```
[draft（編集中）]
    ↓ 確認者が「確認承認」を実行
[confirmed（確認済み）]
    ↓ 確定実行者が「確定して出力」を実行
[finalized（確定済み）]
```

| 状態 | 編集者の操作 | 確認者の操作 | 確定実行者の操作 | ファイル出力 | 印刷 |
|------|-----------|-----------|--------------|-----------|------|
| `draft` | ✅ 全操作可 | ✅ 全操作可 | ✅ 全操作可 | ❌ 不可 | ❌ 不可 |
| `confirmed` | ❌ 編集不可 | ✅ 差し戻し可 | ✅ 確定実行可 | ❌ 不可 | ❌ 不可 |
| `finalized` | ❌ 不可 | ❌ 不可 | ❌ 不可 | ✅ 安全PDFのみ | ✅ 安全PDFのみ |

### 4.3 想定利用者

| 利用者区分 | 対応ロール | IT リテラシー想定 |
|-----------|-----------|-----------------|
| **一般行政職員（担当者）** | 編集者 | 中程度（Officeは使える） |
| **管理職・決裁者** | 確認者 / 確定実行者 | 低〜中程度 |
| **情報システム担当** | （管理者）アプリの導入・保守担当 | 高い |

### 4.4 主要ユースケース

#### UC-01: 情報公開請求対応

1. **編集者**が対象PDFをアプリに読み込む
2. アプリが自動でPIIや機密情報の候補を検出・ハイライト表示する
3. **編集者**が検出結果を確認し、マスキング箇所のON/OFF・修正・追加を行う（手動確認対象も含む）
4. 設定ファイルを保存し、**確認者**に引き継ぐ
5. **確認者**がマスキング内容を確認し、問題なければ「確認承認」を実行（`confirmed`状態へ）
6. **確定実行者**が最終プレビューで全体を確認し「確定して出力」を押す
7. 安全PDFとして出力され、処理ログが自動保存される

#### UC-02: 会議録の公開版作成

1. **編集者**が会議録PDFを読み込む
2. 発言者氏名・役職等が自動検出される
3. 公開基準に従い手動でマスキング対象を調整する（役職名など手動確認対象を含む）
4. **確認者**の承認を経て、**確定実行者**が確定処理を実行し公開用PDFを出力する

#### UC-03: 過去の処理済みファイルの再確認

1. 保存済みのマスキング設定ファイル（JSON）を読み込む
2. 変更が必要な場合は`draft`状態に差し戻し、修正を行う
3. 再度確認・確定処理を実行して出力する

## 5. 機能要件

### 5.1 ファイル入出力

| ID | 要件 | 優先度 |
|----|------|------|
| F-IO-01 | PDFファイル（バージョン1.4〜2.0）の読み込みができること | 必須 |
| F-IO-02 | 複数ページのPDFを一括処理できること | 必須 |
| F-IO-03 | 確定後の安全PDFを指定ディレクトリへ出力できること | 必須 |
| F-IO-04 | マスキング設定情報をJSONファイルとして保存・読み込みできること | 必須 |
| F-IO-05 | ドラッグ＆ドロップによるファイル読み込みができること | 推奨 |
| F-IO-06 | 読み込み可能なファイルサイズは最大100MB（スキャンA4文書想定） | 必須 |
| F-IO-07 | **暗号化PDFの扱い**: 開封パスワード（user password）付きPDFは、パスワード入力ダイアログを表示した上で受け付けること。権限制御（owner password）のみのPDFも読み込めること。復号できないPDF（不明パスワード等）は「パスワードが正しくありません。パスワードを確認して再度お試しください」の定型メッセージで拒否すること | 必須 |
| F-IO-08 | PDFの暗号化方式はAES-128/AES-256/RC4（40bit/128bit）に対応すること（PyMuPDFがサポートする範囲） | 必須 |
| F-IO-09 | **署名付きPDFの扱い**: 入力PDFにデジタル署名（`/Sig` フィールド、certification signature）が含まれる場合、処理は可能とするが署名は必ず失効すること。入力読み込み時に署名を検出した場合、「このPDFにはデジタル署名が含まれています。黒塗り処理後は署名が無効になります。続行しますか？」の警告ダイアログを表示し、ユーザーが「続行」を選択した場合のみ処理を開始すること。「キャンセル」選択時は読み込みを中止する | 必須 |

### 5.2 レイアウト解析・OCR・bbox出力

OCR処理を「レイアウト解析」「文字認識」「bbox出力」の3段に分離して定義する。各段の責務を明確にすることで、実装の崩れを防ぎ、後工程のマスキング編集との接続を安定させる。

#### 5.2.1 レイアウト解析（Layout Analysis）

| ID | 要件 | 優先度 |
|----|------|------|
| F-LAY-01 | PaddleOCR の Layout Analysis モジュールを使用し、各ページを「段落」「表」「図」「ヘッダ/フッタ」等の領域に分類すること | 必須 |
| F-LAY-02 | 表（Table）として検出された領域は、セル単位でのbbox取得を行うこと | 必須 |
| F-LAY-03 | 図・画像として検出された領域は文字認識をスキップし、手動マスキング対象として一覧に表示すること | 推奨 |
| F-LAY-04 | レイアウト解析の結果（領域種別・bbox）をJSON形式で保持し、後工程に引き渡すこと | 必須 |

#### 5.2.2 文字認識（OCR）

テキスト抽出は入力PDFの特性に応じて2系統の処理経路に分岐する。いずれの経路でも、最終的に同じ構造のbbox出力（Section 5.2.3）を生成する。

**経路A: テキスト抽出経路（デジタルネイティブPDF向け）**

| ID | 要件 | 優先度 |
|----|------|------|
| F-TE-01 | PyMuPDFのテキスト抽出機能により、各ページの文字クワッド（character quad）を取得し、bboxソースとすること | 必須 |
| F-TE-02 | 文字クワッドはPDF point座標で取得し、行単位統合ルール（F-BBOX-05）を適用して統合bboxを生成すること | 必須 |
| F-TE-03 | 回転・フォント装飾（太字・斜体等）を含むテキストについても、PyMuPDFのクワッド情報から正確な矩形を取得すること | 必須 |
| F-TE-04 | PyMuPDFのテキスト抽出に失敗した場合（空結果・例外等）は、OCR経路にフォールバックすること | 必須 |
| F-TE-05 | テキスト抽出経路で取得したbboxの `confidence` は **1.0 固定**とする。OCRのような確率的不確実性がないため、信頼度は最高値とする。これにより後段の自動検出・閾値判定でOCR由来と抽出由来が統一的に扱える | 必須 |

**経路B: OCR経路（スキャンPDF向け）**

| ID | 要件 | 優先度 |
|----|------|------|
| F-OCR-01 | PaddleOCR の文字認識モジュールにより、レイアウト解析で抽出した各領域の日本語テキストを認識すること | 必須 |
| F-OCR-02 | Tesseract をフォールバックとして使用できること（PaddleOCRの信頼スコアが閾値0.5未満の場合、または認識文字数が極端に少ない場合・言語判定が不確実な場合に切替） | 推奨 |
| F-OCR-04 | OCR処理中はページ単位のプログレスを表示すること | 必須 |

**経路判定ルール**

入力PDFの各ページについて、以下の判定で経路を選択する。

1. PyMuPDFでテキストレイヤーの有無を確認する
2. テキストが存在する（1文字以上抽出できる）→ **経路A（テキスト抽出経路）**
3. テキストが存在しない（スキャンPDF等）→ **経路B（OCR経路）**
4. 経路Aで抽出結果が空または例外発生 → **経路Bにフォールバック**

> **実装上の注意**: 経路Aで取得した文字クワッドはすでにPDF point座標であるため、座標変換は不要。経路BのOCR結果はピクセル座標からPDF pointへの変換が必要（付録B参照）。行単位統合ルール（F-BBOX-05）は両経路で共通に適用する。

#### 5.2.3 bbox出力

| ID | 要件 | 優先度 |
|----|------|------|
| F-BBOX-01 | 文字認識結果は、テキスト文字列・bbox・信頼スコアをセットにして構造化JSONとして出力すること | 必須 |
| F-BBOX-02 | bboxの座標はPDF point単位（1 point = 1/72 インチ）を正規座標とし、原点はページ左上とすること（詳細は付録B参照） | 必須 |
| F-BBOX-03 | ページに回転（90°/180°/270°）が含まれる場合は、回転補正を施したうえでbboxを出力すること | 必須 |
| F-BBOX-04 | UIへの表示およびPillow/PyMuPDFでの焼き込みに際し、PDF point座標とピクセル座標の変換をDPI設定に基づいて一元管理すること（付録B参照） | 必須 |
| F-BBOX-05 | 近接するbboxを行単位（同一行内の横並び要素）でグルーピングし、意味単位のマスキング領域として統合すること。統合されたbboxは元の個別bboxを包含する最小外接矩形とする | 必須 |

### 5.5 確定マスキング機能（フェーズ2）

#### 5.5.1 可視テキスト・画像の黒塗り処理

| ID | 要件 | 優先度 |
|----|------|------|
| F-FIN-01 | 「確定して出力」ボタンは `confirmed` 状態かつ確定実行者ロールによる操作時のみ押下可能であること | 必須 |
| F-FIN-02 | 確定処理前に確認ダイアログを表示し、マスキング有効件数・対象ページ数を明示のうえ「この操作は元に戻せません」と警告すること | 必須 |
| F-FIN-03 | 確定処理では、各ページを300dpiでラスタライズ（画像化）したうえでマスキング箇所を黒色矩形で焼き込みすること。**黒塗り矩形はbboxそのままではなく、各辺に対して最低3pt（PDF point単位）のマージンを加算すること**。これはOCR bboxの座標誤差・フォント差による視覚的漏れを防ぐためである | 必須 |
| F-FIN-04 | 焼き込み後の画像を再度PDFに変換して出力すること。PDF生成時の圧縮方式は以下の通りとする。**デフォルトはPNG（可逆圧縮・FlateDecode）**とし、行政文書の可読性・証拠性を最優先する。設定からJPEG圧縮（品質85〜95%）への切替を許可するが、品質下限は85%とする。文字中心の行政文書ではJPEG圧縮による輪郭劣化・にじみが発生しやすいため、可逆圧縮を標準とする | 必須 |
| F-FIN-05 | 出力PDFにはOCR由来・元文書由来のテキストレイヤーが一切含まれないこと | 必須 |
| F-FIN-06 | 確定処理後は元のPDFを上書きせず、`<元ファイル名>_redacted_<YYYYMMDD_HHMMSS>_r<revision>.pdf` の命名規則で別ファイルとして保存すること。同日に同一文書を再処理した場合でも一意性を担保するため、日付に加えて時刻（HHMMSS）と revision 番号を付与する。同名ファイルが既に存在する場合は自動採番（`_r2`, `_r3` ...）し、上書きは一切行わない | 必須 |

可視テキストの黒塗りとは別に、以下のhidden dataを確定処理の一環として明示的に除去すること。除去の抜け漏れはセキュリティ上の重大リスクとなる。

| F-SAN-03 | **添付ファイル** | PDFに埋め込まれたファイル（EmbeddedFiles） | 必須 |
| F-SAN-04 | **フォームフィールド** | AcroFormフィールド・XFA（入力済みデータを含む） | 必須 |

## 7. システムアーキテクチャ

### 7.1 技術方式と実装境界

本システムはTauriフレームワークをベースとしたデスクトップアプリケーションとして実装する。Rust・Python・Vanilla JSの責務境界を以下の通り明確に定義する。低スペックな行政PCでの安定動作を優先し、依存関係は最小限に留める。

> **WebView2の前提**: TauriはWindowsでMicrosoft Edge WebView2を使用する。Windows 11には標準搭載されているが、**Windows 10では別途インストールが必要な場合がある**。インストーラーに WebView2 ランタイムのオフラインインストーラーを同梱し、インストール時に自動セットアップすること。

```

│                     Tauri Desktop App                            │
├──────────────────────┬───────────────────────┬───────────────────┤
│   フロントエンド      │       Rustコア         │ Pythonワーカー    │
│   (Vanilla JS /     │   (Tauri バックエンド)  │ (ローカルサブプロセス)│
│    WebView2)        │                        │                  │
│                     │                        │                  │
│ ・UIレンダリング      │ ・ファイルI/O           │ ・レイアウト解析   │
│ ・仮マスキング       │ ・状態管理              │ ・文字認識（OCR）  │
│   オーバーレイ       │ ・JS⇔Python            │ ・bbox出力        │
│ ・PDF表示(PDF.js)   │   プロセス間通信        │ ・PII検出         │
│ ・ウォーターマーク   │ ・ログ記録              │ ・確定マスキング処理│

└──────────────────────┴───────────────────────┴───────────────────┘
```

**実装境界の決定理由**

| 区分 | 担当処理 | 理由 |
|------|---------|------|
| **Vanilla JS** | UI描画・オーバーレイ・状態表示 | Reactを外しバンドル不要・軽量・起動高速 |
| **Rust（Tauriコア）** | ファイルIO・プロセス管理・ログ・セキュリティ | メモリ安全・性能・ネイティブ統合 |

| **Python配布方法** | PyInstaller等で同梱（standalone実行形式） | 行政PCへのPython個別インストールを不要にする |

### 7.3 処理フロー詳細

#### Step 1: レイアウト解析

```
入力PDF → PyMuPDFでページごとに画像化（300dpi）
→ PaddleOCR Layout Analysisを実行
→ 領域リスト（領域種別・bbox）をJSON出力
```

#### Step 2: 文字認識

```
領域リスト → 各領域に対してPaddleOCR文字認識
→ 信頼スコア < 0.5 の領域はTesseractでリトライ
→ テキスト・bbox・信頼スコアをJSON出力
```

#### Step 3: bbox出力・座標変換

```
OCR結果 → PDF point座標に正規化（付録B参照）
→ ページ回転補正を適用
→ 構造化JSONとして State Manager に引き渡す
```

#### Step 4: PII検出

```
テキスト → 正規表現マッチング（自動検出対象）
       → MeCab形態素解析（氏名NE認識）
→ 検出結果リスト（bbox, type, confidence, source="auto"）を生成
```

#### Step 5: 仮マスキング生成

```
検出結果リスト → フロントエンドへIPCで渡す
→ UIオーバーレイとして黒矩形を各bboxに描画
→ JSON状態ファイルに保存（status: "draft"）
→ 「未確定 - 公開禁止」ウォーターマークをビューに表示
→ ファイル出力・印刷の全導線を無効化
```

#### Step 6: ユーザー編集・承認

```
編集者の操作 → ON/OFFトグル・矩形編集・追加・削除
→ 変更のたびにJSONを自動保存・操作ログを記録
確認者の操作 → 確認承認 → status: "confirmed" に遷移
→ 編集操作を無効化（確認者は差し戻しのみ可能）
```

#### Step 7: 確定処理

```
確定実行者が「確定して出力」押下（confirmed状態のみ可）
→ 確認ダイアログ表示（警告・件数明示）
→ OK確認後（Pythonワーカーに処理委譲）：
    1. ページを1枚ずつ300dpiで画像化（PyMuPDF）— **全ページ同時画像化は禁止**（N-AV-04準拠）
    2. 1ページ分の画像に対してenabledな領域に黒色矩形（bbox+最低3ptマージン）を焼き込み（Pillow）
    3. 焼き込み済み画像にPNG圧縮（FlateDecode、デフォルト）を適用後、PDFページとしてバッファに追加（PyMuPDF）
    4. 画像メモリを解放してから次ページへ（逐次パイプライン処理）
    5. 全ページ処理完了後、hidden dataサニタイズ（メタデータ・注釈・添付・フォーム・JS・ブックマーク・隠しレイヤーを除去）
    6. コピー禁止パーミッションを設定（推奨）
    7. 安全PDFとして出力（status: "finalized"）
    8. 一時ファイルを安全削除
    9. 完了ログを記録
```

## 10. セキュリティ要件

### 10.1 確定PDF（安全PDF）のセキュリティ基準

可視テキストの削除だけでは不十分であり、PDFに含まれる **hidden data（不可視情報）の完全除去**が必須である。以下の基準を全て満たすこと。

#### 10.1.1 検証方法: PDFオブジェクト全走査（正式検証基準）

安全PDFの検証は、外部ツール（ExifTool等）による個別確認ではなく、**PyMuPDFによるPDFオブジェクトグラフ全走査**を正式な検証基準とする。「ツールで見えない」ではなく「オブジェクトグラフ上に存在しない」を合格基準とする。

| 検証項目 | 要件 | 検証方法（PDFオブジェクト全走査） |
|---------|------|---------|
| **テキストレイヤー** | PDFにテキストレイヤーが存在しないこと | PyMuPDFでテキスト抽出を試みて空であること。ページの `/Contents` ストリームにテキスト描画命令（`Tj`, `TJ`, `'`, `"`）が含まれないこと |
| **ページ内容の画像由来性** | ページの可視内容が全て画像XObject由来であること | テキスト描画命令（`Tj`, `TJ`, `'`, `"`）が存在しないこと。画像描画に必要な制御命令（`q`, `Q`, `cm`, `Do`, `rg`, `RG` 等）は許容する。**合格条件は「テキスト命令がないこと」を主条件とし、「Doのみで構成」は要件としない** |
| **メタデータ（DocInfo）** | DocInfo辞書の全キーが空または除去されていること | trailer辞書の `/Info` エントリが存在しない、または全値が空文字列であること |
| **メタデータ（XMP）** | XMPメタデータストリームが存在しないこと | `/Metadata` エントリが存在しないこと |
| **注釈** | `/Annots` 配列が存在しない、または空であること | ページ辞書の `/Annots` を確認 |
| **添付ファイル** | `/EmbeddedFiles` が存在しないこと | カタログ辞書の `/Names` → `/EmbeddedFiles` を確認 |
| **フォームフィールド** | `/AcroForm` が存在しないこと | カタログ辞書の `/AcroForm` を確認 |
| **JavaScript** | `/JS`, `/OpenAction`, `/AA` エントリが存在しないこと | カタログ辞書およびページ辞書の該当キーを確認 |
| **ブックマーク** | `/Outlines` が存在しない、または空であること | カタログ辞書の `/Outlines` を確認 |
| **隠しレイヤー** | `/OCProperties` が存在しないこと | カタログ辞書の `/OCProperties` を確認 |
| **増分更新** | 増分更新セクションが存在しないこと | PDF末尾の `%%EOF` が1つのみであることを確認 |
| **コピー禁止** | PDFパーミッション設定でテキスト選択・コピーが禁止されていること | PDFパーミッションフラグを確認（推奨） |

#### 10.1.2 サニタイズ失敗時の取扱い

検証の結果、いずれかの項目が不合格となった場合、**出力PDFを破棄しエラーとして処理を中断**すること。不完全なPDFをユーザーに渡すことはセキュリティ上許容されない。エラー詳細はログに記録し、ユーザーには「処理に失敗しました。管理者に連絡してください」のメッセージを表示する。

#### 10.1.3 開発時の二重チェック

開発フェーズでは、PyMuPDFによるPDFオブジェクト全走査に加え、ExifTool / pdftotext / pdfimages 等の外部ツールによる二重確認を行い、検証ロジックの正しさを担保すること。外部ツールは開発時の検証補助であり、本番の合格判定には使用しない。

### 10.2 アプリケーションセキュリティ

| 項目 | 要件 |
|------|------|
| ローカル処理 | 入力PDF・処理データは全てローカルで処理し、外部ネットワークに送信しないこと |
| 一時ファイル | 処理中に作成した一時ファイル（画像化中間ファイル等）は確定処理完了後に安全削除すること |
| ログの秘匿 | ログファイルに元の個人情報テキストを記録しないこと（region_id・bbox・typeのみ記録） |
| ファイルアクセス | アプリが必要とするディレクトリ以外へのファイルアクセスを行わないこと（Tauriのallowlistで制限） |

### 10.3 仮マスキング状態の流出防止（状態遷移制約）

ウォーターマークは視覚的抑止力であり保護機構ではない。以下を**アプリケーションレベルの状態遷移制約**として実装すること。

| 状態 | 制限内容 |
|------|---------|
| `draft` / `confirmed` | ファイルメニューの「保存」「名前を付けて保存」「エクスポート」「共有」を無効化 |
| `draft` / `confirmed` | 印刷（メニュー・Ctrl+P・右クリック印刷）を無効化 |
| `draft` / `confirmed` | ドラッグ&ドロップによる外部へのファイルドロップを無効化 |
| `draft` / `confirmed` | ファイルをエクスプローラーに表示する操作をブロック（中間ファイルのパスを非公開） |
| `finalized` | 安全PDFのファイルパスのみ公開し、元PDFおよび設定ファイルのパスは非公開 |

## 11. エラー処理・例外要件

| エラー種別 | 検出条件 | 対応 |
|-----------|---------|------|
| ファイル読み込み失敗 | PDFが破損 | エラーダイアログを表示し、処理を中断する。「ファイルが破損しているか、対応していない形式です」 |
| パスワード保護PDF | PDFが暗号化されている | パスワード入力ダイアログを表示。復号成功時は通常処理に進む。復号失敗時は定型エラーメッセージを表示（F-IO-07参照） |
| OCR処理失敗 | エンジンエラー・タイムアウト | Tesseractでリトライ。それも失敗した場合はユーザーに通知し、手動マスキングのみで継続可能とする |
| 確定処理失敗 | 画像変換失敗・書き込み権限なし | エラーダイアログを表示し、元PDFは変更しない。ログにエラー詳細を記録する |
| 設定ファイル破損 | JSON解析エラー | バックアップから復元を試みる。失敗した場合はエラーを通知し、新規状態で継続する |
| ディスク容量不足 | 出力先のディスク空き容量が処理に必要な推定容量を下回る | 処理前に容量チェックを行い、不足する場合は警告を表示する |

## 12. ログ・監査要件

### 12.1 ログ保存先・保存期間

- **保存先**: アプリのデータディレクトリ（例: `%APPDATA%\RedactionApp\logs\`）
- **ファイル名**: `redaction_YYYYMMDD.jsonl`（日次ローテーション）
- **保存期間**: 最低3年間（行政文書の保管基準に準拠する運用とする）
- **最大サイズ**: 1ファイルあたり100MB（超過時は圧縮アーカイブ）

### 12.2 監査対応

- ログは改ざん防止のため、アプリからは読み取り専用でアクセス可能とする
- **ハッシュチェーン方式の採用**: 各ログレコードに前レコードのSHA-256ハッシュ値を含め、レコード間の改ざん検知を可能にすること。これにより、ログの部分的な削除・改ざんが後から検出可能となる（監査対応で必須となる整合性確認要件）
- **日次ルートハッシュの別保管**: 日次ログファイルのクローズ時（日次ローテーション時）に、最終レコードのハッシュ値を「ルートハッシュ」として別ファイル（例: `%APPDATA%\RedactionApp\logs\root_hashes.jsonl`）に追記保存すること。ログファイル本体が丸ごと差し替えられても、別保管されたルートハッシュとの不一致で検出可能とする。ルートハッシュファイルはアプリからは追記のみ可能とし、OS ACLで管理者のみ読み取り可能に設定すること
- **操作者IDの必須記録**: 全ログレコードの `user` フィールドには、固定値 `"system"` ではなく**実際の操作者識別子**（OSログイン名＋再入力表示名）を必ず記録すること
- 情報公開請求の対応記録として、案件IDと紐付けたログのエクスポート機能を提供する（将来対応）

### 13.1 ベースライン端末仕様

性能要件は以下のベースライン端末仕様で評価する。この仕様を下回る端末での動作は保証しない。

### 13.2 処理時間要件

| 処理 | 対象 | 目標時間（ベースライン端末） | 最悪ケース許容時間 |
|------|------|---------|---------|
| OCR処理 | A4 1ページ（スキャン300dpi） | 5秒以内 | 10秒以内 |
| OCR処理 | A4 10ページ | 60秒以内 | 120秒以内 |
| テキスト抽出（デジタルPDF） | A4 1ページ | 1秒以内 | 3秒以内 |
| 自動検出 | 10ページ分のOCR結果 | 3秒以内 | 5秒以内 |
| 仮マスキング描画 | 1ページあたり50件 | 0.5秒以内 | 1秒以内 |
| 確定処理 | A4 10ページ | 30秒以内 | 60秒以内 |
| PDF表示 | ページ切替 | 1秒以内 | 2秒以内 |

### 13.3 UI応答性の最低保証

- OCR・確定処理等の長時間処理はバックグラウンドスレッドで実行し、**UIスレッドをブロックしないこと**
- 処理中のキャンセル操作に応答すること（キャンセル要求から1秒以内にUIが反応すること）
- プログレスバーは全体進捗を百分率で表示し、10秒以上更新がない場合は「処理が停止している可能性があります」を表示すること

**補足**: OCRがボトルネックとなるため、バックグラウンドスレッドで実行しUIをブロックしないこと。

## 14. 制約・前提条件

### 14.1 技術的制約

- 動作OS: Windows 10 / 11（64bit）に限定する（macOS・Linuxは対象外）
- 確定処理方式は「画像化→黒塗り焼き込み→PDF再生成」を採用する（PDFネイティブテキスト削除方式は実装難易度・リスクの観点から採用しない）
- OCRはローカル実行とし、クラウドOCRは使用しない

### 14.2 前提条件

- 入力PDFは日本語文書を主対象とする
- スキャン解像度が150dpi以下の場合はOCR精度が低下する可能性があり、この場合は手動マスキングで補完する運用を想定する
- 導入先環境にインターネット接続は不要とするが、初回インストール時のみは管理者権限が必要となる可能性がある

## 付録A: 設計判断の根拠

### A.1 確定方式：画像化方式を採用する理由

PDFのテキストをネイティブに削除する方式（方式B）は技術的に可能だが、PDFの内部構造（フォント埋め込み・圧縮ストリーム・参照構造等）が複雑であり、実装の不具合によってテキストが残存するリスクが排除しきれない。

これに対し、**画像化→黒塗り焼き込み→PDF再生成方式（方式A）** は、原理的にテキストが存在しえない構造のPDFを生成するため、最も安全性が高く実装も安定している。

行政文書という性質上、安全性を最優先とし方式Aを採用する。

### A.2 2段階設計を採用する理由

即座に不可逆処理を行う設計では、誤ってマスキングすべき箇所を見逃した場合や、マスキング対象を誤って選択した場合のリカバリーが不可能となる。行政実務では「複数人での確認」「上長への報告」といったフローが存在するため、編集可能な状態での中間保存と不可逆確定の分離は必須要件である。

## 付録B: 座標系・DPI仕様

### B.1 正規座標系の定義

本システムは **PDF point（1 pt = 1/72 インチ）** を唯一の正規座標単位とする。全コンポーネント（UI表示・設定ファイル保存・確定焼き込み）間の座標受け渡しは、すべてPDF pointで行う。

- **原点**: ページ左上 (0, 0)
- **X軸**: 右方向に増加
- **Y軸**: 下方向に増加
- **単位**: PDF point（浮動小数点）
- **bbox表現**: `[x, y, width, height]`

### B.2 座標変換の単一正規化関数

全座標変換は **単一の正規化関数**（Python側に実装し、IPC経由でJS側にも座標を渡す）により行う。各コンポーネントでの個別実装は禁止し、実装側の座標変換ミスを構造的に防止する。

#### B.2.1 PDF point → 表示ピクセル（UI用）

```
px = pt * (display_dpi / 72)
```

- `display_dpi`: WebViewの表示DPI（OS設定に基づく実効DPI）
- 丸め: 四捨五入（`round`）

#### B.2.2 表示ピクセル → PDF point（UI入力→保存用）

```
pt = px * (72 / display_dpi)
```

- 丸め: 小数点第3位まで保持（切り捨て: `floor` at 3 decimal places）

#### B.2.3 PDF point → ラスタライズピクセル（焼き込み用）

```
raster_px = pt * (raster_dpi / 72)
```

- `raster_dpi`: 設定ファイルの `dpi_for_rasterize`（デフォルト300）
- 丸め: 切り上げ（`ceil`）— 黒塗り漏れ防止のため、焼き込み座標は必ず外側に広げる

### B.3 ページ回転時の座標補正（bbox単位）

PDFのページ回転（0°/90°/180°/270°）が設定されている場合、bbox座標は回転補正後の値とする。点座標の変換だけでなく、**bbox（矩形全体）の変換**として定義する。

#### B.3.1 bboxの回転補正手順

bbox `[x, y, width, height]` を左上点・右下点に分解し、2点を個別に変換したうえで再正規化する。

```
入力: bbox = [x, y, width, height]
左上点: (x1, y1) = (x, y)
右下点: (x2, y2) = (x + width, y + height)

Step 1: 各点を回転変換
Step 2: 変換後の2点から min/max で正規化bboxを再構成
Step 3: 90°/270° の場合は width/height を入れ替え
```

#### B.3.2 回転角度ごとの変換式

**回転 0°（そのまま）**
```
bbox_out = bbox_in
```

**回転 90°（時計回り）**
```
新左上点: (y1, page_width - x2)
新右下点: (y2, page_width - x1)
bbox_out = [min(y1,y2), min(page_width-x2, page_width-x1), |y2-y1|, |x2-x1|]
※ width/height が入れ替わる
```

**回転 180°**
```
新左上点: (page_width - x2, page_height - y2)
新右下点: (page_width - x1, page_height - y1)
bbox_out = [page_width-x2, page_height-y2, width, height]
```

**回転 270°（反時計回り）**
```
新左上点: (page_height - y2, x1)
新右下点: (page_height - y1, x2)
bbox_out = [page_height-y2, x1, |y2-y1|, |x2-x1|]
※ width/height が入れ替わる
```

#### B.3.3 保存座標の正規形

- `page_width`, `page_height`: 回転前のページ寸法（PDF point）
- 設定ファイルには **回転補正後の正規化済み bbox** を保存する
- UI・焼き込みでは保存値をそのまま使用し、再度の回転補正は行わない
- 補正はテキスト抽出・OCRの結果保存時に一度だけ適用する

### B.4 変換の適用方向

座標変換は **一方向（正規座標→各用途座標）** のみを許可し、逆変換による累積誤差を防止する。

```
[PDF point（正規）] ─→ 表示ピクセル（UI）
[PDF point（正規）] ─→ ラスタライズピクセル（焼き込み）

ユーザー操作（ドラッグ等）→ PDF pointに変換して保存 → 再表示時は保存値から再計算
```
